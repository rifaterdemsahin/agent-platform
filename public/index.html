What is the Agent Platform?

A multi-agent runtime where each agent is a self-contained unit with its own spec, code, and skills. Agents run as background processes, exposed over a REST API, and powered by LLMs (Claude via Anthropic SDK).

What agents exist

AgentRoleSkillsTriggers

What each agent contains

agents/<name>/
  ├── agent.json          # SPEC: name, version, description, skills[], runtime, triggers[]
  ├── index.js            # CODE: agent entrypoint — exports { agent, run() }
  └── skills/             # SKILLS: standalone modules callable by name
      └── skill-name.js

What the runtime does

runtime/agent-loader.js scans the agents/ directory at startup, parses every agent.json, and imports each agent's code. The Express server exposes:

  • GET /api/agents — list all agents
  • GET /api/agents/:name — get agent spec
  • GET /api/agents/:name/skills — list skills
  • POST /api/agents/:name/invoke — execute a skill
  • GET /api/skills — all skills across all agents

Agent = spec + code + skills in one folder

No YAML config sprawl. No separate service per agent. One Express server loads every agent from the filesystem. Add a folder, push to GitHub, GitHub Actions deploys to Fly.io — the new agent is live.

Why an agent platform?

Problem

Every project needs the same background checks — linting, type safety, env validation, cost tracking, deployment readiness. Building these from scratch per project wastes time and leads to inconsistency.

Why not separate services?

  • Fragmented — each service has its own deploy, config, auth, monitoring
  • Inconsistent — different APIs, different error shapes, different logging
  • Hard to discover — no central registry of what agents exist and what they can do
  • Duplicate LLM plumbing — every service reinvents the Anthropic integration

The agent platform treats agents like npm packages — self-contained, versioned, discoverable, and invocable over HTTP. But instead of publishing to a registry, you push to a monorepo and the runtime picks them up.

How to iterate on the Agent Platform

How to add a new agent

  1. Create agents/<name>/ folder
  2. Create agent.json (name, version, description, skills[], runtime, triggers[])
  3. Create index.js exporting { agent, run() }
  4. Create skill modules in skills/
  5. git add agents/<name>/ && git commit && git push origin main

GitHub Actions auto-deploys to Fly.io. The agent is live on the next API poll.

How to update an agent's skills

  1. Edit agents/<name>/skills/<skill>.js
  2. If adding a new skill, add to skills[] in agent.json
  3. Add the skill case in run() in index.js
  4. Push to GitHub — Fly.io redeploys automatically

How to call agents from other projects

// JavaScript
const res = await fetch("https://agent-platform-9a3f.fly.dev/api/agents/sanity-check/invoke", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ skill: "deploy-readiness", params: { path: "." } })
});

// Python
import requests
r = requests.post(
  "https://agent-platform-9a3f.fly.dev/api/agents/budget/invoke",
  json={"skill":"estimate-cost","params":{"model":"claude-sonnet-4-20250514","inputTokens":5000,"outputTokens":2000,"provider":"anthropic"}}
)

How deployment works

1. Push
To main on GitHub
2. CI
GitHub Actions triggers deploy.yml + pages.yml
3. Build
flyctl deploy builds Docker image
4. Ship
Fly.io rolls out new machines (zero-downtime, 2 instances)
5. Verify
Health check hits /api/agents every 30s
6. Pages
GitHub Pages deploys static dashboard from public/
